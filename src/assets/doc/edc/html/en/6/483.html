Documentation types

Documentation types cater to different information requirements. 

By defult, edc proposes:

  • Contextualised UI Help: the basic building block of contextualised help
  • How To: short procedures for accomplishing tasks
  • Concepts: definitions and explanations about concepts
  • Training: step-by-step instructions for learning to use your software
  • Q&A: anticipate user questions, can be qualified as FAQ later
  • Glossary: short definitions of unfamiliar terms.

You can define your own Documentation types, in Configuration > Content management > Strategies.

If necessary, you can also give your documentation type an alias.

Contextualised UI help

This is a map of the product UI bricks.

Usually, the map is provided by a developer. Writers can also build the brick map.

Content for contextualised help should be brief. Keep the end-product in mind: the user help appears in the software.

How tos 

These are simple steps for using a part of the product and carrying out one simple task.

Steps are grouped into larger procedures that make sense in terms of what the user wants to do, not only based on feature know-how.

Concepts

A concept is used to explain abstract information about your product. Concepts can be grouped into themes.

Background information can be used where necessary without saturating the user. For example, they may simply need to be reminded how to carry out a task without procedures.

Training

Design these as short explanations with real data that can be read by the user in-site or offline.

One specific use is gettings users up and running.

Writing for this type of information requires an understanding of user personas and how to write for these personas.

Questions and answers

Forget the notion of FAQ. Questions have answers. Analytics will determine if they are frequently read.

Glossary

Use this to explain unfamiliar terms to your users. The definition appears as a pop-up.
